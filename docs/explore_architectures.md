### Observations

#### Model Capability & Behavior
- **Haiku (Low Capability):** Consistently failed to meet goals; demonstrated alignment issues by ignoring instructions and entering infinite loops of aimless exploration (aborted one test after 14 minutes).
- **Fable (High Capability):** Succeeded after some exploration, confirming that a more capable LLM is required for these tasks.

#### Execution & Integration Bottlenecks
- **Connection & Login Flow:** The agent struggled with the deterministic MUD login flow and lacked visibility into failed attempts. Increasing model capability did not resolve this core connection issue.
- **Inefficient Resource Usage:** The agent repeatedly created ad-hoc scripts to manage socket connections and read unrelated files, resulting in unnecessary token usage.

### Technical Conclusions

- A coding harness alone is not a good fit for operating the MUD.

- The deterministic login and connection flow should be handled by a reusable MUD SDK rather than regenerated by the model on every attempt.

- A specialized agent loop, potentially using an MCP server connected to the MUD SDK, would provide better reliability, visibility, and token efficiency.
